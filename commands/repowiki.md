---
description: Convert any code repository into a rich Obsidian wiki using multi-agent analysis
allowed-tools: Bash(mkdir:*), Bash(rm -rf:*), Bash(git diff:*), Bash(git log --oneline:*)
---

Convert the current repository into an Obsidian-compatible wiki by orchestrating a pipeline of specialized sub-agents. You are the **orchestrator** — you do NOT analyze code directly. You coordinate agents and pass information between them.

## Usage Examples

```
python cli.py --section "Database"   # regenerate one section only
python cli.py --dry-run              # preview plan without generating
python cli.py --depth shallow        # fastest: ~90% fewer input tokens
python cli.py --depth medium         # default: ~73% fewer tokens, includes method previews
python cli.py --depth deep           # full source code (no compression)
```

## Step 0: Parse Arguments

The user may provide arguments. Parse them from the command args:

- `--output <dir>` — output directory (default: `.repowiki`)
- `--lang <en|pt>` — wiki language (default: `en`)
- `--section "<name>"` — generate only one section by title (optional)
- `--update` — only regenerate sections whose files changed since last run

Store these as variables you'll use throughout:
- `OUTPUT_DIR` = parsed value or `.repowiki`
- `LANG` = parsed value or `en`
- `SECTION_FILTER` = parsed value or null
- `UPDATE_MODE` = true if `--update` was passed

If `UPDATE_MODE` is true, run `git diff --name-only HEAD` to get changed files. You'll use this later to filter which sections to regenerate.

Report to the user: `Starting repowiki... output → <OUTPUT_DIR>/`

## Step 1: Setup

```bash
mkdir -p "<OUTPUT_DIR>/.tmp"
mkdir -p "<OUTPUT_DIR>/_meta"
```

## Step 2: Discovery Phase

Spawn a sub-agent with `subagent_type: "repowiki:discovery"` and this prompt (fill in the actual paths):

---
**Discovery agent prompt:**

```
You are the Discovery Agent for repowiki.

Repo root: <current working directory>
Manifest output path: <OUTPUT_DIR>/.tmp/_manifest.json

Scan the repository. Ignore: .git, .venv, .repowiki, node_modules, __pycache__, migrations, staticfiles, dist, build, coverage, .tmp

Use Glob to find all source files by type. Read the first 30 lines of these files if they exist: README.md, package.json, requirements.txt, go.mod, Cargo.toml, pyproject.toml, config/settings.py, config/settings.py.example, app.py, main.py, main.go, index.js, src/index.ts.

Write the manifest JSON to: <OUTPUT_DIR>/.tmp/_manifest.json

Return MANIFEST_COMPLETE when done.
```
---

After the discovery agent completes, read `<OUTPUT_DIR>/.tmp/_manifest.json` and store the manifest content.

## Step 3: Architecture Phase

Spawn a sub-agent with `subagent_type: "repowiki:architect"` and this prompt:

---
**Architect agent prompt:**

```
You are the Architect Agent for repowiki.

Read the discovery manifest at: <OUTPUT_DIR>/.tmp/_manifest.json

Design the wiki structure for this repository. Consider its complexity (<complexity from manifest>) and detected patterns (<detected_patterns from manifest>).

Write the architecture JSON to: <OUTPUT_DIR>/.tmp/_architecture.json

Return ARCHITECTURE_COMPLETE when done.
```
---

After the architect agent completes, read `<OUTPUT_DIR>/.tmp/_architecture.json` and parse the sections array.

If `SECTION_FILTER` is set, filter `sections` to only those whose `title` matches (case-insensitive).

If `UPDATE_MODE` is true, filter `sections` to only those whose `key_files` overlap with the git-changed files list.

## Step 4: Create Section Directories

For each section in the (possibly filtered) sections array:
```bash
mkdir -p "<OUTPUT_DIR>/<section.output_dir>"
```

## Step 5: Parallel Specialist Generation

**CRITICAL: Spawn ALL specialist agents IN PARALLEL by making multiple Agent tool calls in a single response.**

For EACH section, spawn a sub-agent with `subagent_type: "repowiki:specialist"` with this prompt (customized per section):

---
**Specialist agent prompt (one per section, all spawned simultaneously):**

```
You are a Specialist Wiki Agent for repowiki.

Section: <section.title>
Output directory: <OUTPUT_DIR>/<section.output_dir>/
Language: <LANG>
Context: <section.context>

Read these files completely (use the Read tool on each):
<section.key_files — one per line>

Generate these wiki pages:
<section.pages — one per line>

Write each page as a .md file to <OUTPUT_DIR>/<section.output_dir>/<Page Name>.md

Follow the specialist agent format exactly:
- YAML frontmatter with tags, category, wiki_version, generated, sources
- # Page Title (H1)
- ## Table of Contents with [[WikiLinks]]
- Content sections with > **Sources:** attribution after each
- At least one Mermaid diagram per page
- Footer: *[[index|← Back to Index]] · Generated by repowiki*

Every technical claim must be grounded in the files you read. Never invent class names, fields, or endpoints.

Return SPECIALIST_COMPLETE: <section.title> — N pages written
```
---

Wait for ALL specialist agents to complete before proceeding.

## Step 6: Finalization Phase

Spawn a sub-agent with `subagent_type: "repowiki:finalizer"` and this prompt:

---
**Finalizer agent prompt:**

```
You are the Finalizer Agent for repowiki.

Output directory: <OUTPUT_DIR>/
Repository name: <repo_name from manifest>
Total sections: <count>

Use Glob to find all .md files in <OUTPUT_DIR>/ (exclude .tmp/ and _meta/).

1. Build an inventory of all pages: { "Category/Page Name": path }
2. Read every page and check all [[WikiLinks]] — fix any that point to non-existent pages
3. Write <OUTPUT_DIR>/index.md — the vault entry point with full navigation table
4. Write <OUTPUT_DIR>/_meta/repowiki-metadata.json — UUID-based knowledge graph

Return FINALIZATION_COMPLETE with summary stats.
```
---

## Step 7: Cleanup

```bash
rm -rf "<OUTPUT_DIR>/.tmp"
```

## Step 8: Report to User

After the finalizer completes, report:

```
✓ repowiki complete

Output:    <OUTPUT_DIR>/
Sections:  <n>
Pages:     <m>

Open in Obsidian:
  Settings → Open vault → select <OUTPUT_DIR>/

Start at:  <OUTPUT_DIR>/index.md
```

If any agent failed or returned an error, report which phase failed and the error message, then stop.
